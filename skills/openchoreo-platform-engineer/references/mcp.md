# OpenChoreo MCP

Single source of truth for OpenChoreo's two MCP servers: tool catalog, workflow patterns (scaffold a component, build, deploy, debug, deploy a multi-service third-party app), and every gotcha. Used by all OpenChoreo skills as their MCP reference тАФ no per-skill duplicates.

| Server | Tool prefix | Purpose |
|---|---|---|
| Control plane | `mcp__openchoreo-cp__*` | Resource CRUD, schemas, workflows, releases |
| Observability | `mcp__openchoreo-obs__*` | Logs, metrics, traces, spans, alerts, incidents |

Both must be configured separately; neither covers the other's surface.

> **Prefer MCP over `occ` for the operations in this catalog.** MCP wins for resource CRUD, schema discovery, and observability queries тАФ structured payloads, no shell quoting, typed responses. Use `occ` when you don't see an MCP tool here for the operation, or for the inherently CLI-only flows listed in `cli.md` (login, version, context management, `apply -f`, scaffold). The "CLI Equivalent" column below is a fallback, not the recommendation.

## Exploration Workflow

When connecting to an unfamiliar cluster, explore in this order:

```
1. list_namespaces
2. list_projects(namespace)
3. list_environments(namespace)
4. list_deployment_pipelines(namespace)
5. list_cluster_component_types       тЖТ cluster-wide types (most common)
6. list_component_types(namespace)    тЖТ namespace-scoped types (if any)
7. list_cluster_traits                тЖТ cluster-wide traits
8. list_cluster_workflows             тЖТ cluster-scoped build workflows
9. list_workflows(namespace)          тЖТ namespace-scoped workflows (if any)
10. list_components(namespace, project) тЖТ components already deployed
```

## Workflow Patterns

### 1. Scaffold and Create a Component

Discover what's available before authoring YAML:

```
list_cluster_component_types       тЖТ find available workload types (e.g. deployment/service)
get_cluster_component_type_schema  тЖТ inspect fields and options for a type
list_cluster_traits                тЖТ find available traits (e.g. ingress, storage)
get_cluster_trait_schema           тЖТ inspect trait fields
get_component_schema               тЖТ get the full Component resource schema
```

Then create:

```
create_component(namespace, project, name, spec_yaml)
```

For BYO image deployments, **omit `workflow` from the component spec**.

### 2. Build (Trigger Workflow)

```
list_workflows(namespace)                               тЖТ what workflows are available?
list_workflow_runs(namespace, project, component)       тЖТ check existing run history
trigger_workflow_run(namespace, project, component)     тЖТ kick off a build
get_workflow_run(namespace, project, component, run_id) тЖТ check run status
```

### 3. Activate and Manage Deployments

After a build completes, release bindings are created automatically (or on first `create_workload` for BYO):

```
list_release_bindings(namespace, project, component)           тЖТ see binding per environment
get_release_binding(namespace, binding_name)                   тЖТ inspect status, endpoints, state
update_release_binding_state(namespace, binding_name, Active)  тЖТ activate a deployment
update_release_binding_state(namespace, binding_name, Undeploy)тЖТ undeploy from an environment
patch_release_binding(namespace, binding_name, overrides)      тЖТ update env overrides or release ref
```

> Deployment is via `create_release_binding` (added in v1.0.0-rc.2). There is no separate `deploy_release` or `promote_component` tool. To promote to a downstream environment, create a new ReleaseBinding for that environment via `create_release_binding` (or `occ apply -f releasebinding.yaml`), then call `update_release_binding_state` with `Active`.

### 4. Inspect and Debug

```
get_component(namespace, project, component)         тЖТ spec + status conditions
list_workloads(namespace, project, component)        тЖТ running workloads
get_workload(namespace, workload_name)               тЖТ workload details
list_release_bindings(namespace, project, component) тЖТ binding per environment
get_release_binding(namespace, binding_name)         тЖТ binding status, endpoints, invokeURL
list_secret_references(namespace)                    тЖТ available secrets
```

### 5. Query Logs and Metrics (observability MCP)

`query_component_logs` is the fastest way to triage runtime behavior from the AI assistant. `query_workflow_logs` covers build/CI logs.

```
query_component_logs(namespace, project, component, environment)   тЖТ runtime logs
query_workflow_logs(namespace, project, component, workflow_run)   тЖТ build logs
query_http_metrics(namespace, project, component, environment)     тЖТ request rate / latency / errors
query_resource_metrics(namespace, project, component, environment) тЖТ CPU and memory
```

Trace debugging:

```
query_traces(namespace, project, component, environment) тЖТ find relevant traces
query_trace_spans(trace_id)                              тЖТ list spans in a trace
get_span_details(span_id)                                тЖТ inspect a single span
```

### 6. Create or Update a Workload

The right tool depends on whether the component uses a build workflow:

**BYO-image component** (no `spec.workflow`) тАФ developer creates the workload:
```
get_workload_schema(namespace, project, component)         тЖТ understand workload fields first
create_workload(namespace, project, component, name, spec) тЖТ create workload
list_workloads(namespace, project, component)              тЖТ verify it appears
```

**Source-build component** (`spec.workflow` set) тАФ workload is auto-generated by the build (named `{component}-workload`):
```
trigger_workflow_run(namespace, project, component)        тЖТ kick off the build
list_workloads(namespace, project, component)              тЖТ confirm {component}-workload appears after the build
get_workload(namespace, '{component}-workload')            тЖТ inspect what the build produced
update_workload(namespace, '{component}-workload', spec)   тЖТ enrich (endpoints, deps, env, files) ONLY if repo has no workload.yaml
```

**Preferred path for enriching source-build workloads**: edit `workload.yaml` in the source repo and rebuild. The build re-creates the workload with the descriptor inlined. Reach for `update_workload` only when you can't or don't want to rebuild.

> `update_workload` takes only `namespace_name`, `workload_name`, `workload_spec` тАФ not `project_name` or `component_name`. Use `list_workloads` to find the workload name first.

> The auto-generated workload is **always** named `{component}-workload`, even if `workload.yaml` declares a different `metadata.name` тАФ the build overrides it. So `occ workload get my-svc` returns "not found"; use `occ workload get my-svc-workload`.

### 7. Deploy a Third-Party / Multi-Service App with Pre-built Images

When the user asks to deploy a well-known public or open-source multi-service app:

```
# 1. Find pre-built images тАФ check release/ directory, README, CI pipelines
WebFetch or gh to fetch official kubernetes-manifests.yaml (or Helm values, docker-compose)
   тЖТ extract image URLs and ALL env vars per service

# 2. Create project
create_project(namespace, name)

# 3. Create all components тАФ NO workflow parameter
create_component(namespace, project, name, componentType)   тЖТ repeat for each service
   componentType examples:
     deployment/service         тЖТ backend gRPC/REST/TCP services
     deployment/web-application тЖТ public-facing frontend
     deployment/worker          тЖТ background workers, load generators
     statefulset/datastore      тЖТ stateful stores (Redis, databases)

# 4. Apply workloads in batch
# Batch 1: workloads without dependencies (simpler, fewer failure modes)
# Batch 2: workloads with dependencies
occ apply -f /tmp/<app>-workloads.yaml
occ apply -f /tmp/<app>-workloads-dependencies.yaml

# 5. Verify each component
list_release_bindings(namespace, component)  тЖТ Ready or ResourcesProgressing?

# 6. Investigate any failing component immediately тАФ don't assume platform issue
query_component_logs(namespace, project, component, environment)
   тЖТ crash before "listening on port"? тЖТ vendor SDK crash, missing env var, or startup panic
   тЖТ connection refused from another service? тЖТ dependency not yet ready or wrong port/env var
```

**Key rules for this workflow:**

- Never set `workflow` in `create_component` for BYO image deployments.
- Always extract ALL env vars from official manifests тАФ dependencies inject service addresses but not `PORT`, feature flags, or vendor SDK disable flags.
- If a service crash-loops before logging a "listening" message, look for a native module load error or vendor SDK init failure тАФ apply the disable flag from the official manifests.
- If a required env var references an optional or not-yet-deployed service, set a placeholder value to prevent startup panics.
- `dependencies` is an **object containing an `endpoints` array** тАФ entries live under `dependencies.endpoints[]`, with `name` for the target endpoint (not `endpoint`).
- Source builds fail for repos that use `ARG BUILDPLATFORM` multi-stage syntax (exit code 125) тАФ switch to BYO immediately when you see this error.

## Gotchas

**Component type format is `workloadType/typeName`**: Use `get_cluster_component_type_schema` to see accepted values before setting `spec.componentType.name`.

**`get_component` and `get_workload` include status.conditions**: Each condition has `type`, `status`, `reason`, `message`. Always check conditions when debugging.

**`list_release_bindings` requires both project and component**: You must pass both, not just project.

**`get_workload_schema` before writing workload YAML**: Call this to discover the field shape rather than guessing. `dependencies` is an object with an `endpoints` array тАФ see the gotcha below.

**`dependencies` is nested**: The Workload `dependencies` field is an **object** containing an `endpoints` array, not a flat array. Each entry uses `name` (target endpoint name), not `endpoint`:
```yaml
dependencies:
  endpoints:
    - component: backend-api
      name: api                # the target endpoint name on backend-api
      visibility: project       # project | namespace
      envBindings:
        address: BACKEND_URL
```
Field renamed from `connections` in v1.0.0.

**`trigger_workflow_run` vs `create_workflow_run`**: `trigger_workflow_run` starts a build for a component using its configured workflow (pass optional `commit` SHA to pin to a revision). `create_workflow_run` creates a standalone run by workflow name with explicit parameters тАФ use for workflows not tied to a component.

**Workflow runs can lag**: A just-triggered workflow may briefly show no runs. Call `list_workflow_runs` after a moment, then verify with `get_component`.

**Two separate MCP servers**: `mcp__openchoreo-cp__*` (control plane) and `mcp__openchoreo-obs__*` (observability) тАФ both must be configured.

**`update_workload` only takes `namespace_name`, `workload_name`, `workload_spec`**: It does not accept `project_name` or `component_name`. Use `list_workloads` to find the workload name first.

**Observability data requires a live ObservabilityPlane**: If queries return no data, the plane may be missing or unhealthy. Escalate to `openchoreo-platform-engineer`.

**TCP `address` binding injects `host:port`, not a protocol DSN**: For databases (PostgreSQL, MySQL) and message brokers (NATS, Redis), the injected `address` value is a plain `host:port` string. Apps that expect `postgres://user:pass@host/db` or `nats://host:4222` will fail to parse it. Declare the dependency for the topology diagram but set the full DSN as a literal env var instead. Get the hostname from `get_release_binding` тЖТ `endpoints[*].serviceURL.host`.

**Source-build workloads have no endpoints or dependencies until you add them**: The `generate-workload-cr` build step creates a minimal workload with just the image. If the repo has no `workload.yaml`, always call `update_workload` after a successful build to add endpoints, env vars, and dependencies. Without this the component deploys but has no routing, no dependencies, and the cell diagram renders incomplete.

**File mount `mountPath` is a directory**: The controller appends the `key` name to `mountPath` to form the final file path. Set `mountPath` to the parent directory (`/usr/share/nginx/html`), not the full file path (`/usr/share/nginx/html/config.json`). Using the full file path doubles the filename: `.../config.json/config.json`.

**Browser-facing apps need `https://` and `wss://` backend URLs**: OpenChoreo serves web-application components over HTTPS. Any backend URLs injected at runtime (e.g., via a mounted `config.json`) must use `https://` and `wss://`. HTTP/WS URLs are blocked by browsers as mixed content тАФ no visible error, requests just silently fail. Always get external URLs from `get_release_binding` тЖТ `endpoints[*].externalURLs` and use the `https` scheme.
