# GitOps with OpenChoreo

OpenChoreo supports a GitOps workflow where all platform and application resources live in a Git repository, with Flux CD syncing changes to the cluster.

## Repository structure

```
gitops-repo/
├── flux/                              # Flux CD orchestration
│   ├── gitrepository.yaml             # Points to this repo
│   ├── namespaces-kustomization.yaml
│   ├── platform-shared-kustomization.yaml
│   ├── platform-kustomization.yaml    # Depends on namespaces + shared
│   └── projects-kustomization.yaml    # Depends on platform
│
├── namespaces/
│   └── <namespace>/
│       ├── namespace.yaml             # With openchoreo.dev/controlplane-namespace label
│       ├── platform/                  # PE-owned
│       │   ├── component-types/
│       │   ├── traits/
│       │   ├── workflows/
│       │   └── infra/
│       │       ├── environments/
│       │       └── deployment-pipelines/
│       └── projects/                  # Team-owned
│           └── <project>/
│               ├── project.yaml
│               └── components/
│                   └── <component>/
│                       ├── component.yaml
│                       ├── workload.yaml
│                       ├── releases/          # Generated by workflows
│                       └── release-bindings/  # Generated by workflows
│
└── platform-shared/                   # Cluster-wide resources
    └── cluster-workflow-templates/
```

## Flux dependency chain

Resources must apply in order:

```
namespaces → platform-shared (ClusterWorkflowTemplates) → platform (types, traits, envs) → projects (components, releases)
```

Each Kustomization declares `dependsOn` to enforce ordering.

## GitOps release workflows

### Single-component build and release

The `docker-gitops-release` workflow:

1. Clones source repo, builds image, pushes to registry
2. Extracts `workload.yaml` from the source
3. Generates a ComponentRelease YAML (frozen snapshot)
4. Generates ReleaseBinding YAML for the root environment
5. Creates a PR in the GitOps repo with the new files

Merging the PR triggers Flux to apply the release.

### Bulk promotion

The `bulk-gitops-release` workflow promotes all components in a project (or all projects) to a target environment:

1. Reads existing releases
2. Generates ReleaseBinding YAMLs for the target environment
3. Creates a single PR with all bindings

Useful for promoting an entire project from staging to production in one step.

## Setting up GitOps

### Prerequisites

- Flux CD installed on the cluster
- A Git repository for the GitOps state
- Git credentials stored as a Kubernetes secret for Flux
- Git credentials stored in the secret backend for workflow access (so build workflows can create PRs)

### Flux GitRepository

```yaml
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: openchoreo-gitops
  namespace: flux-system
spec:
  interval: 1m
  url: https://github.com/your-org/your-gitops-repo
  ref:
    branch: main
  secretRef:
    name: git-credentials
```

### Workflow configuration

GitOps workflows need a parameter pointing to the GitOps repo so they can create PRs:

```yaml
parameters:
  gitops:
    repoUrl: 'string'
    branch: 'string | default=main'
    basePath: 'string | default=.'
```

The ClusterWorkflowTemplate handles cloning, generating release YAMLs, committing, and pushing via PR.

## PE responsibilities in GitOps

**Owns**: `flux/`, `namespaces/*/platform/`, `platform-shared/`

**Reviews**: PRs to `namespaces/*/projects/` (component definitions, release promotions)

**Automates**: Build workflows create PRs for releases. Bulk workflows create PRs for promotions. PEs review and merge.
